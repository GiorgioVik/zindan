# Публикация релиза на GitHub

Каждый релиз на GitHub получает **краткое описание изменений** автоматически.

## Перед тегом

1. Обновите **`CHANGELOG.md`**: добавьте секцию с номером сборки, например:

   ```markdown
   1.5.4 (244)
   ===

   **APK:** `Zindan-1.5.4-(244)-debug.apk`

   - Краткий пункт: что изменилось для пользователя
   - Ещё один пункт
   ```

2. Закоммитьте и отправьте в `main`.

3. Создайте и отправьте тег (номер в теге = `VERSION_CODE`):

   ```bash
   git tag v1.5.4-244
   git push origin v1.5.4-244
   ```

GitHub Actions соберёт APK и опубликует [Release](https://github.com/GiorgioVik/zindan/releases) с текстом из CHANGELOG.

## Откуда берётся текст релиза

| Источник | Когда |
|----------|--------|
| `RELEASE_NOTES_v1.5.4-244.md` | Если файл есть — используется он (для особых релизов) |
| **`CHANGELOG.md`** | Секция `(244)` — **основной способ** |
| Заглушка | Только если секции нет |

## Необязательно

Файлы `RELEASE_NOTES_v*.md` нужны только если для релиза нужен текст **отличный** от CHANGELOG (например, длинное полевое резюме). Для обычных сборок достаточно CHANGELOG.
