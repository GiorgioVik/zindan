Тестовые устройства: Samsung Galaxy S24 Ultra (SM-S928B/DS), Tab S9 FE+ (SM-X616B).

## Частично решено в 1.3.8

### Anti Spy: VPN + работающее приложение в work profile

**Статус:** полуавтоматический обход (без Shizuku)  
**Затронуто:** Zindan 1.3.8+

**Было (1.3.7):** фоновый монитор видел VPN, но автозаморозка из `:vpnwatch` в work profile не доставлялась.

**Стало (1.3.8):** при VPN-up — короткий dummy-VPN (вытеснение туннеля ~500 мс), затем уведомление:
«Закройте приложение в work profile» → **Да** = групповая заморозка (как снежинка); **Нет** = ничего (VPN Zindan не восстанавливает, приложения работают).

**Ограничения:**
- Zindan **не убивает** процесс в work — пользователь закрывает приложение сам.
- Пока приложение не закрыто, оно может снова увидеть VPN (если клиент переподключился).
- «Нет» подавляет повторный запрос до отключения VPN.
- Без разрешения VPN у Zindan dummy-цикл не снимает туннель — показывается диалог **«Нужен доступ к VPN»** или **«Не удалось прервать VPN»** (1.3.10: на экране, не в шторке), а не «Да / заморозить всё».

**Полная автозаморозка без участия пользователя** по-прежнему требует Shizuku или иной привилегированный путь.

### Сценарий 2 — открытый пробел (приоритет следующей итерации)

**Условие:** приложение из списка **автозаморозки** уже в work profile (на экране или в фоне) → пользователь **включает VPN**, Anti Spy (золотой щит) **включён**.

| Ожидание | Факт в 1.3.10 |
|----------|----------------|
| «Заморозить всё» (как снежинка) | **Не вызывается** автоматически |
| Хотя бы диалог «Да / Нет» | **Почти никогда** — цепочка рвётся до UI |

**Техническая причина (аудит кода):**

- `AntiSpyVpnWatchService` **видит** VPN (те же классы сигналов, что MacroDroid `VpnTrigger`).
- Реакция: `maybeFreezeAllForVpn()` → dummy-VPN → `showPrompt()` **только** если туннель снят и в момент показа VPN снова не активен.
- `AntiSpyManager.freezeAllForVpn()` (фоновая групповая заморозка) **нигде не вызывается** при VPN-up.
- Категории уведомлений «Anti Spy: VPN» / «Zindan: важное» **не заменяют** разрешения **VPN** и **поверх других приложений** для основного профиля.

**Обход до патча:** после включения VPN вручную нажать снежинку / ярлык групповой заморозки.

**Не путать** с другим сценарием (запуск приложения при уже работающем VPN) — тот путь **работает** и в baseline **не менять** без необходимости.

## Отложено

### Galaxy Store — не клонируется

**Статус:** отложено (как в upstream Shelter)  
**Затронуто:** Zindan, **оригинальный Shelter** — одинаковое поведение

**Симптом:** клонирование Galaxy Store из основного профиля в рабочий не работает (тихий отказ / сообщение о системном приложении). Play Store и остальные приложения клонируются нормально.

**Причина (предположительно):** пакет с флагом system, в work profile ещё не установлен; upstream использует только `enableSystemApp()`, чего недостаточно для OEM-магазинов.

**В Zindan3:** логика `installApp` = **upstream Shelter** (экспериментальный фикс `needsPackageInstallerClone` откатан — давал краш).

**Пакет:** `com.sec.android.app.samsungapps`

**Когда вернёмся:**

- общее решение для system + не установлен в профиле (не хардкод Samsung);
- разбор `logcat`, если снова будет краш;
- запасной путь через `installApk` (как в Zindan2).

## Отложено — баги после 185 (следующая итерация, не в 185)

**Статус:** зафиксировано при тесте / обсуждении, **не чинить в 185**  
**Тег для отката:** `v1.5.2-build185`

### 1. Сообщение «Все приложения заморожены» — внизу (шторка)

**Симптом:** после групповой заморозки текст «Все приложения в списке автозаморозки были успешно заморожены» появляется **внизу** в шторке (уведомление рабочего профиля, иконка с портфелем), тогда как остальные сообщения Anti Spy — **вверху**.

**Ожидание:** все пользовательские сообщения только **сверху** (toast ~6 с), без дублирующего уведомления в рабочем профиле.

**Направление фикса (черновик 187):** убрать `postVpnAutoFreezeSuccessAlert` при ручной/ярлыковой групповой заморозке; оставить только `showToastOnMainProfile`.

### 2. Стартовая групповая заморозка при загрузке

**Симптом:** после перезагрузки / открытия Zindan срабатывает «Заморозить всё», даже когда **все уже заморожены**; показывается лишнее сообщение; возможно **прерывание VPN**.

**Ожидание:** при boot/первом запуске замораживать только **ещё не скрытые** приложения из списка; если менять нечего — тишина; не мешать Anti Spy VPN.

**Направление фикса (черновик 187):** считать только вновь замороженные; стартовая заморозка через **фон** (`runStartupFreeze`), не UI-цепочку `DummyActivity`.

### 3. Клонирование основной → рабочий: двойной «Установить приложение?»

**Симптом:** при «Клонировать в Zindan» диалог подтверждения установки появляется **дважды**; иногда первый тап не срабатывает (блокировка запросом уведомлений в рабочем профиле).

**Ожидание:** один диалог или **без диалога** (через `installExistingPackage` на API 28+).

**Направление фикса (черновик 190):** `installExistingPackage` + guard `PENDING_USER_ACTION` + не блокировать install запросом POST_NOTIFICATIONS.

### 4. Ярлыки групповых операций — пересоздать после 185

**Симптом:** старые ярлыки на рабочем столе ведут в `DummyActivity` (поведение до 185).

**Действие для теста:** удалить старые → создать заново из меню Zindan / Настройки.

### 5. Снежинка на ярлыках запуска (группа D)

**Статус:** **откачено** по решению пользователя в 181; в 185 **нет**. Не тестировать до отдельного запроса.
